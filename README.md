# PipelineRL: Faster On-policy RL for Long Sequence Generation

Implementation of the [PipelineRL paper](https://arxiv.org/abs/2509.19128) (Piché, Kamalloo, Pardinas, Chen & Bahdanau, 2025).

## What is PipelineRL?

Conventional reinforcement learning for LLMs alternates between two phases: first generate a full batch of sequences, then train on them. This means GPUs sit idle during one phase or the other — a major bottleneck when generating long reasoning chains that can take minutes per sequence.

PipelineRL eliminates this bottleneck by running generation and training **concurrently on separate GPUs**. The key innovation is **in-flight weight updates**: the inference engine receives updated model weights *during* ongoing token generation, without stopping or restarting. This creates "mixed-policy" sequences where early tokens were generated by an older policy and later tokens by the updated policy. The paper demonstrates ~2x faster learning compared to conventional RL while keeping data highly on-policy.

## Key Highlights from the Paper

### 1. In-flight Weight Updates (Section 4, Algorithm 2, Lines 9-11)

The core innovation. The vLLM actor receives new weights between token generation steps via `reload_weights()`. It briefly pauses (~0.4s) to reload, then continues generating — no restart, no KV cache loss, no idle time. Sequences currently being generated get the new weights for their **remaining** tokens.

### 2. Concurrent Generation and Training (Figure 1b)

Unlike conventional RL which serializes `generate → train → generate`, PipelineRL runs both simultaneously. The Actor continuously generates at constant batch size H while the Trainer continuously consumes completed sequences. Neither process ever waits for the other.

### 3. Importance-weighted REINFORCE (Eq. 5)

Since sequences may have been generated by an older behavior policy μ, the loss uses truncated importance sampling weights `min(c, π_θ/μ)` to correct for off-policyness. The clamp c=5 prevents high-variance gradient estimates from stale data.

For verifiable tasks like GSM8K (binary reward: 1 if correct, 0 otherwise), no learned value function is needed. The baseline is simply the mean reward in the batch (GRPO-style group relative baseline), which is equivalent to Eq. 5 with v_φ replaced by the batch mean reward.

### 4. Mixed-policy Sequences (Figure 3a)

After an in-flight update, sequences in progress become mixed-policy: early tokens from the old policy, later tokens from the new policy. The paper shows (Figure 7) that these mixed-policy sequences stay close to on-policy as measured by KL divergence, while fully off-policy sequences diverge significantly.

### 5. Token Lag Tracking (Figure 3a, Figure 6a)

Each token has a "lag" — the number of optimizer steps between when it was generated and when it's used for training. In PipelineRL, the lag varies *within* a sequence: early tokens have higher lag, later tokens have lower lag (because they benefited from more recent in-flight updates). This is tracked via `start_weight_version` and `end_weight_version` per sequence.

### 6. Effective Sample Size (ESS) Monitoring (Eq. 6, Figure 6b)

ESS = (Σw_i)² / (N·Σw_i²) measures how on-policy the training data is. ESS near 1.0 means data is fresh and importance weights are uniform; low ESS signals stale off-policy data that could destabilize training. The paper shows PipelineRL maintains ESS comparable to conventional RL with G=8.

### 7. Constant Batch-size Inference (Section 4)

The Actor always maintains H sequences in progress. As sequences complete, new prompts are immediately submitted to keep the batch full. This ensures high GPU utilization regardless of variable sequence lengths — unlike conventional RL where the batch size shrinks as shorter sequences finish (Figure 2b).

## Architecture

```
    GPU 0 (Actor / vLLM AsyncLLMEngine)           GPU 1 (Trainer)
   ┌──────────────────────────────┐              ┌──────────────────────────────┐
   │                              │  completed   │                              │
   │  Continuous batching engine  │  sequences   │  HuggingFace model           │
   │  - H concurrent requests    │  (pickle)    │  - Collect B sequences       │
   │  - Non-blocking async gen   │ ───────────> │  - REINFORCE loss (Eq. 5)    │
   │  - Per-token log probs      │              │  - Mean-reward baseline      │
   │                              │  updated     │    (no value function)       │
   │  In-flight weight update:   │  weights     │  - Optimizer step            │
   │  - reload_weights() called  │  (safetens)  │  - publish_weights()         │
   │  - Sequences keep generating│ <─────────── │                              │
   │  - Remaining tokens use     │              │  Logs: reward, loss, ESS,    │
   │    NEW weights (mixed-policy)│              │  token lag, mixed-policy cnt │
   └──────────────────────────────┘              └──────────────────────────────┘
```

**Communication**: Filesystem-based IPC (robust with vLLM's internal subprocess architecture).
- Actor → Trainer: Completed sequences as pickle files, including per-token log probs
- Trainer → Actor: Model weights in HuggingFace safetensors format, reloaded via vLLM's `collective_rpc_async('reload_weights')`

## Setup

Requires 2 GPUs. Tested on 2x NVIDIA RTX A6000 (48GB).

```bash
# Install all dependencies
uv sync
```

## Training

```bash
# Quick smoke test (3 optimizer steps, ~1 min)
uv run python scripts/run.py --config configs/test.yaml

# Full training run (500 optimizer steps)
uv run python scripts/run.py --config configs/default.yaml
```

### Logged Metrics (per step)

| Metric | Description | Paper Reference |
|---|---|---|
| `reward` | Mean reward in the batch | Figure 5a |
| `loss` | IS-weighted REINFORCE loss | Eq. 5 |
| `ESS` | Effective Sample Size (on-policyness) | Eq. 6, Figure 6b |
| `token_lag [min..max]` | Range of per-token lag in the batch | Figure 3a, Figure 6a |
| `avg_token_lag` | Average lag of oldest tokens per sequence | Figure 6a |
| `mixed_policy` | Count of sequences with in-flight updates | Section 4 |
| `seq/s` | Throughput (sequences per second) | Figure 5c |

Metrics are saved to `outputs/metrics.json`. Checkpoints saved every 50 steps.

## Key Design Decisions

1. **No value function**: GSM8K rewards are binary and verifiable (correct/incorrect), so a learned critic adds complexity without benefit. The batch mean reward serves as a sufficient baseline (GRPO-style).

2. **AsyncLLMEngine over LLM**: Using vLLM's async engine gives true mid-sequence in-flight updates. Multiple generation requests run concurrently in vLLM's continuous batching engine. When `reload_weights()` is called, all in-progress sequences get new weights for their remaining tokens.

3. **Filesystem IPC over multiprocessing Queue**: vLLM spawns its own EngineCore subprocess, making Python multiprocessing Queue unreliable. Pickle files for results + safetensors for weights are robust across process boundaries.

4. **Separate OS process for Actor**: Ensures CUDA_VISIBLE_DEVICES isolation. The Actor subprocess has full control of GPU 0, the Trainer (main process) has GPU 1.
